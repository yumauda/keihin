## 2. xd-code/ の解析

### 2-1. ファイル構造とJSON形式

各アートボードは **`{番号}_001__{Adobe XDのオブジェクトID連番らしき数字}__{UUID}.png`** + **`export_{番号}.json`** のペアで書き出されている（一部 `001___` でアンダースコアが2つになっているのは群番号無しのケース）。

**JSON のスキーマ**（1行JSON）:
```json
{
  "artboards": [
    {
      "name": "...",
      "type": "text" | "group",
      "bounds": { "width": ..., "height": ..., "x": ..., "y": ... },
      "style": { "fontFamily": ..., "fontSize": ..., "fontStyle": ..., "textAlign": ..., "lineHeight": ..., "fill": { "rgba": "...", "hex": "#..." } },
      "id": "uuid",
      "transform": { "a": 1, "b": 0, "c": 0, "d": 1, "e": x, "f": y },
      "text": "...",
      "children": []
    }
  ]
}
```

- `type: "text"` の場合 → そのアートボードはテキスト1要素を切り出したもの。`text` プロパティでテキスト内容、`style` で `fontFamily / fontSize / lineHeight / fill.hex` 等が完全に取れる
- `type: "group"` の場合 → 複合要素（カード群・SVGアイコン・写真合成等）を1グループとして書き出したもの。詳細は付随PNGで確認するしかない（`children` は配列だが実質空のことが多い）
- `bounds.x / bounds.y` はカンプ全体内の絶対座標、`bounds.width / bounds.height` は要素自身のサイズ
- **重要**: 各JSONには `artboards` 配列があるが、確認した全17ファイルとも **配列長は1**（= アートボード = 切り出し要素1つ）

## 4. Phase 2（画像配置）への引き継ぎ事項

### 4-1. 結論: **新規に xd-code/ → src/images/ に追加配置すべきファイルは原則ない**

**理由**: TOPページ9セクションで使う画像は `src/images/common/` `src/images/top/` に既に揃っており、`images/` への圧縮 + WebP 生成も済んでいる。xd-code/ のPNGは「カンプの構図確認用の切り出し」であって「実装に使う画像素材」ではない。

### 4-2. ただし以下は要確認・要対応

| 対応 | 内容 | 担当 |
|---|---|---|
| **確認**（Phase 2不要 / Phase 3に直接渡してOK）| 既存 `src/images/` 内の画像が、最新カンプと差分がないか目視確認（特に MV / 代表写真 / CAREER写真 / Instagramカード）| Phase 3 担当が必要に応じて確認 |
| **削除候補** | `src/images/.DS_Store`（無害だが整理推奨）| Phase 3 担当 or 別途 |
| **対応必要（小）** | `src/images/common/` に **`top-instagram-sp.png`** が無い（`src/images/top/top-instagram-sp.png` は存在）。`index.php` は `./images/top/top-instagram-sp.webp` を参照しているので **現状動いている** が、配置パスを揃えるかは判断必要 | Phase 3 担当 |
| **新規必要が出たとき** | カンプ修正で新規画像を追加する場合 → **必ず `src/images/{common or top}/` に PNG/JPG で配置 → `npm run dev` で自動圧縮&WebP生成** | 全フェーズ共通ルール |

### 4-3. 画像配置の鉄則（Phase 3 含めて全担当に再周知）

1. **ソース画像の置き場所は必ず `src/images/`**（`images/` 直下に手で置かない）
2. **拡張子はオリジナル（PNG / JPG / SVG / GIF）**で配置 → ビルドが自動で WebP も生成
3. **`<img>` の参照先は `./images/...`**（書き出し後のパス）
4. **`<img>` には `decoding="async" loading="lazy" width="N" height="N"` を必ず付与**（既存コードがその流儀）
5. **WebP優先・PNG/JPG はフォールバック**。`<picture>` で出し分けする場合の例は `index.php` の `.p-top-welfare__instagram` を参照

```scss
  padding-top: rem(20);

  @include mq(md) {
    // PC（768px以上）
    font-size: rem(40);
    padding-top: rem(96);
  }
}
```

- **1セクション = 1ファイル = `_p-{name}.scss`**
- 新規追加時は `src/sass/styles.scss` の `// Project` ブロックに `@use "./project/_p-{name}";` を **アルファベット順** で追記
- **すべて `rem()` 関数経由**で px → rem 変換（`rem(40)` のように使う、16px基準）
- メディアクエリは **`@include mq(md)`** のみで十分なケースが多い（必要に応じて `sm / lg / xl` も）

### 5-5. HTMLの定型ルール

- 画像: `<img decoding="async" loading="lazy" src="./images/..." alt="..." width="N" height="N">` を **必ず** 守る
- MV のような LCP 対象画像のみ `loading="lazy"` を外す（参考: `.p-mv__img img` には `loading="lazy"` が付いていない＝意図的）
- セクションは `<section class="p-{name}">` で開き、内側で `<div class="l-inner">` → `<div class="p-{name}__content">` の入れ子が定型
- PHP インクルードは `<?php include_once($_SERVER['DOCUMENT_ROOT'] . '/includes/{name}.php'); ?>` の絶対パス形式（**Local等のローカルサーバー前提**で動く書き方なので、別環境では DocumentRoot 設定に注意）

### 5-6. JSの流儀

- jQuery + GSAP（CDN）+ 自前 `js/script.js` `js/gsap.js`
- `src/main.js` は事実上未使用 → **既存通り `js/` ディレクトリに直接書く**
- GSAP は `gsap.timeline()` に `fromTo` / `to` を順番に積む書き方（読みやすさ優先 / 過度な抽象化はしない）
- 1文字ずつ表示は **SplitText 不使用**、HTML / JS で `<span>` に分割 + `stagger` で出す
- jQuery のフックは **`js-` プレフィックスのクラス**で取る（CSS装飾とJSフックを分離）
